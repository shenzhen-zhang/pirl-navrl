# EGO Official Sidecar Baseline Path

## Current TASK_02 Status

TASK_02 has completed the official EGO-Planner Docker/ROS sidecar diagnostic
route. The active route runs official EGO planner/controller/simulator nodes in
the Noetic Docker sidecar and uses PIRL-NavRL scripts only for scene injection,
trace recording, and PyBullet mirroring.

The current diagnostic scene suite can inject custom pointcloud obstacles into
the official EGO ROS loop through `/pirl_navrl/custom_scene_cloud`, remapped to
EGO's `/grid_map/cloud`. The implemented scene modes are:

- `ego_static_obstacle_v0`: fixed custom obstacle pointclouds.
- `ego_dynamic_obstacle_v0`: continuously moving custom obstacle pointclouds.
- `ego_sudden_motion_obstacle_v0`: initially stationary, then moving custom
  obstacle pointclouds.

PyBullet is still a mirror in this route. It visualizes the same obstacle
primitives and selected official ROS topics, but it is not the
gym-pybullet-drones control backend and does not close the simulation-control
loop.

This route is diagnostic only. It is not a formal baseline, does not produce
baseline metrics, and should not be reported as a paper result.

## Required Work Before Baseline Use

To promote this diagnostic route into an EGO baseline, the project still needs:

- EGO command to gym-pybullet-drones action integration.
- A shared `ScenarioConfig` used by both the EGO sidecar and native PyBullet
  evaluation route.
- Unified metrics generated by the same evaluator used for other methods.
- Explicit collision and success definitions.
- Multi-seed evaluation with fixed scenario generation and reporting protocol.
- Additional ROS graph/topic checks to confirm the exact map-consumption path
  inside the official EGO nodes.

Until those pieces exist, TASK_02 should be described as an official EGO
sidecar diagnostic and baseline-readiness spike, not as a baseline result.
